Day 7 Lean Principles & Waste Elimination

Lean methodology focuses on maximizing value by eliminating waste and optimizing processes. This approach is crucial for PMP exam success and real-world project management.

üéØ Core Lean Principle

Lean = Eliminate anything that doesn't add value + Keep what's valuable

Every time you see "Lean," think: Remove waste, maximize value

üóëÔ∏è The 7 Types of Waste (Muda)

These are critical for the PMP exam. Know them cold!

Waste Type Definition Examples
1. Partially Done Work Incomplete features that can't be used Code waiting for testing, specs waiting for development
2. Extra Processes Unnecessary documentation or steps Excessive meetings, redundant approvals, over-documentation
3. Extra Features Building more than customer requested Gold plating - adding features customer didn't ask for
4. Task Switching Constantly changing between tasks Multitasking, context switching, interruptions
5. Waiting Idle time, delays, bottlenecks Waiting for approvals, dependencies, resources
6. Motion Unnecessary movement of people/information Excessive handoffs, poor workspace layout
7. Defects Errors requiring rework Bugs, quality issues, miscommunication

‚ö†Ô∏è Critical Exam Distinction

Gold Plating vs. Scope Creep

Gold Plating:

  • Project team voluntarily adds extra features
  • Customer did NOT request these features
  • Done to "impress" or "delight" customer
  • Example: Customer wants cardboard ceiling, you install expensive gypsum
  • PMI View: ‚ùå BAD - Can cause cost overruns and delays

Scope Creep:

  • Uncontrolled expansion of project scope
  • Changes made without formal change control
  • Unauthorized additions to requirements
  • Stakeholders keep changing requirements informally
  • Result: Missed deadlines, cost overruns, increased risk

Key Difference: Gold plating is giving extras; scope creep is accepting unauthorized changes

üö´ Task Switching: The Hidden Productivity Killer

üí° Why Task Switching is Waste

  • Context Switching Cost: Takes 15-30 minutes to refocus after interruption
  • Quality Suffers: More errors when multitasking
  • Productivity Drop: Up to 40% productivity loss
  • Increased Stress: Mental fatigue from constant switching

Lean Solution: Focus on completing one task before starting another (single-piece flow)

üîÑ Hybrid Approaches

Day 7 also introduced combining Agile and Predictive methodologies:

  • Predictive-Agile Hybrid: Use waterfall for planning, agile for execution
  • Use Cases: Regulated industries, fixed budgets with flexible scope
  • Benefits: Combines structure with flexibility
  • Challenges: Requires careful integration of both approaches

üìù Exam Focus Areas

  • Memorize all 7 types of waste
  • Clearly distinguish gold plating from scope creep
  • Understand why Lean eliminates waste
  • Know when to use hybrid approaches
  • Recognize task switching as a major waste

üéØ Lean in Action: Real-World Examples

‚úÖ Lean Success Example

Scenario: Software development team struggling with delays

Lean Analysis:

  • Identified waste: Excessive documentation (extra processes)
  • Identified waste: Features waiting for testing (partially done work)
  • Identified waste: Team members juggling 5+ tasks (task switching)

Lean Solution:

  • Reduced documentation to "just enough"
  • Implemented continuous integration/testing
  • Limited WIP to 2-3 items per person

Result: 40% faster delivery, 50% fewer defects

üí∞ Project Selection Techniques

Organizations must decide which projects to pursue. Here are the key financial techniques:

1. Benefit-Cost Ratio (BCR)

Formula: BCR = Total Benefits / Total Costs

Decision Rule: BCR > 1 = Project is worthwhile

Example: Benefits = $500K, Costs = $300K ‚Üí BCR = 1.67 ‚úÖ

When to use: Quick evaluation of project viability

2. Payback Period

Definition: Time needed to recover initial investment

Formula: Payback Period = Initial Investment / Annual Cash Flow

Decision Rule: Shorter payback = Better project

Example: Investment = $100K, Annual return = $25K ‚Üí Payback = 4 years

Limitation: Doesn't consider time value of money

3. Return on Investment (ROI)

Formula: ROI = (Net Profit / Investment) √ó 100%

Decision Rule: Higher ROI = Better project

Example: Investment = $200K, Net Profit = $80K ‚Üí ROI = 40%

When to use: Comparing profitability across projects

4. Net Present Value (NPV) üî•üî•üî•

üí° NPV is CRITICAL for the exam!

Definition: Present value of future cash flows minus initial investment

Formula: NPV = PV of Benefits - Initial Investment

Decision Rule:

  • NPV > 0 = Project adds value ‚úÖ
  • NPV = 0 = Project breaks even
  • NPV < 0 = Project loses money ‚ùå
  • Choose project with HIGHEST NPV

Why it matters: Accounts for time value of money (money today worth more than tomorrow)

Project NPV Decision
Project A $150,000 ‚úÖ Good
Project B $250,000 ‚úÖ‚úÖ Best Choice!
Project C -$50,000 ‚ùå Reject

5. Internal Rate of Return (IRR)

Definition: Discount rate that makes NPV = 0

Decision Rule: Higher IRR = Better project

Comparison: IRR should exceed organization's required rate of return

Example: If IRR = 15% and required return = 10%, project is good ‚úÖ

6. Opportunity Cost

‚ö†Ô∏è Critical Concept for Exam

Definition: Value of the project you DIDN'T choose

Example:

  • Project A NPV = $300K (chosen)
  • Project B NPV = $200K (not chosen)
  • Opportunity Cost = $200K (what you gave up)

Exam Tip: Opportunity cost = Value of NEXT BEST alternative

üõ†Ô∏è Tools & Techniques for Developing Project Charter

Creating the project charter requires specific tools and techniques to gather the right information:

1. Expert Judgment üî•üî•üî•

Definition: Knowledge and experience provided by people who have expertise in preparing charters, the project industry, or organizational processes.

Why It's Critical:

  • Project sponsors may not have technical knowledge of the industry
  • Charter components (objectives, requirements, metrics) need specialized input
  • Design and implementation teams provide essential insights
  • Business analysts help define broad requirements

Sources of Expert Judgment:

  • Business analysts for requirements definition
  • Technical experts from design/implementation teams
  • Industry specialists with domain knowledge
  • Organizational leaders with process knowledge

2. Meetings

Meetings are essential for:

  • Gathering expert judgment - Experts are met in the context of meetings
  • Data gathering - Information not found in business documents
  • Collaboration - Team alignment and consensus building

3. Data Gathering

Technique Description Best Use
Brainstorming Team members collectively generate ideas Charter components needing creative input
Focus Groups Specialized groups discuss specific areas Different sections of project charter
Interviews One-on-one discussions with experts Gathering specific technical knowledge

üí° Focus Groups vs Interviews

  • Focus Groups: More conversational, less formal than interviews
  • Focus Groups: Allow document analysis and collective opinions
  • Focus Groups: Multiple perspectives on same topic
  • Interviews: More structured, individual insights

4. Interpersonal and Team Skills

Critical skills for working with experts and stakeholders:

  • Conflict Management - People have differing opinions; bring them to alignment
  • Facilitation - Leading meetings effectively
  • Meeting Management Techniques:
    • Prepare agenda in advance
    • Ensure right people are represented
    • Send timely invites
    • Prepare and share minutes
    • Document follow-up action items

üìã Developing the Project Management Plan

The project management plan is a comprehensive document that defines the basis of all work to be done and outlines how the work will be performed.

üéØ Charter vs Plan

Aspect Project Charter Project Plan
Detail Level Summary information Detailed, comprehensive
Purpose Authorizes project Defines how work will be done
Structure Single document Multiple integrated sub-plans

üìä The 18 Sub-Plans (9-4-3-2 Mnemonic) üî•üî•üî•

The project management plan consists of 18 sub-plans organized by the 9-4-3-2 mnemonic:

9 Area Plans (Knowledge Area Plans)

One plan for each knowledge area EXCEPT Integration Management:

  1. Scope Management Plan - How to define and control scope
  2. Schedule Management Plan - How to develop and control schedule
  3. Cost Management Plan - How to plan and control costs
  4. Quality Management Plan - Quality standards, metrics, and tools
  5. Resource Management Plan - How to acquire and manage resources
  6. Communications Management Plan - What, when, how to communicate
  7. Risk Management Plan - How to identify and manage risks
  8. Procurement Management Plan - Contract types, make/buy decisions
  9. Stakeholder Engagement Plan - How to engage stakeholders

4 Baselines (Approved for Control)

‚ö†Ô∏è Critical Exam Concept: Baselines

Baselining in PMI = Approving for Control

Baseline What It Is Measured Against
Schedule Baseline Approved project schedule Actual progress
Cost Baseline Approved project budget Actual costs
Scope Baseline Approved project scope Actual deliverables
Performance Measurement Baseline Technique to measure project health Schedule, cost, scope performance

3 Scope-Related Plans

  1. Requirements Management Plan
    • Process for gathering requirements
    • Documenting requirements
    • Making changes to requirements
  2. Configuration Management Plan
    • Making changes to project components as result of one change
    • Assessing impact of changes
    • Updating all affected configurable areas
  3. Change Management Plan ‚ö†Ô∏è
    • NOT about organizational change (transition management)
    • IS about change control - scope, cost, schedule changes
    • Process from proposal to implementation
    • Managing variations in the project

‚ö†Ô∏è Change Management: Two Meanings

Organizational Change Management (NOT in PMP scope):

  • Managing transition from current to future state
  • Changing organizational culture
  • Getting people to adopt project changes

Project Change Management (IN PMP scope):

  • Change control for scope, cost, schedule
  • Managing change requests / variations
  • Guidelines from proposal to implementation

2 Lifecycle Plans

  1. Project Life Cycle Description
    • Listing of project phases
    • Description of each phase
    • Overview of project flow from start to end
    • Especially important for waterfall projects
  2. Development Approach
    • Method to evolve product from start to end
    • Examples: Waterfall, Agile, Hybrid
    • Explains how product will be delivered
    • Framework description (e.g., "Scrum with 2-week sprints")

üéØ Tailoring the Project Plan

‚ú® Tailoring Concept

PMI allows you to include only plans important for YOUR project!

  • Don't have to include all 18 plans
  • Don't have to include everything defined in each plan
  • Adjust depth and breadth based on:
    • Project size
    • Project scope
    • Project complexity
  • Only include what's important for your specific project

üé™ Kickoff Meetings

Kickoff meetings communicate objectives, gain commitment, and explain roles. Timing varies by project size:

Project Type Kickoff Timing Rationale
Large/Complex Projects AFTER creating project plan ‚Ä¢ Use small team for planning
‚Ä¢ Large teams make planning inefficient
‚Ä¢ Planning team solicits input from larger team
‚Ä¢ Once plan complete, engage all stakeholders
Small Projects BEFORE planning (after charter) ‚Ä¢ One team does planning AND execution
‚Ä¢ Small teams can plan efficiently together
‚Ä¢ No harm in engaging everyone early

üí° Kickoff Meeting Purpose

Regardless of timing, kickoff meetings serve to:

  • Communicate project objectives
  • Gain commitment from the team
  • Explain roles and responsibilities
  • Align stakeholders on project approach

üéØ Agile Projects: Planning Differences

üìù Agile Project Plans

Agile projects typically do NOT have all 18 plans!

What Agile Projects DO have:

  • Development Approach - Framework description (e.g., Scrum)
  • Schedule Baseline - Known at start (ballpark figure)
  • Cost Baseline - Budget ceiling
  • Project Charter - Defines boundaries, user stories

What Agile Projects DON'T have:

  • Scope Baseline - Scope is dynamic!
  • Most area plans - Due to dynamic scope
  • Detailed schedule/budget - Can't plan with dynamic scope

Instead of elaborate plans: Agile depends on project charter and product backlog to provide boundaries

üìä Data ‚Üí Information ‚Üí Reports

Understanding the progression from data to reports is critical for the exam:

1. Work Performance Data (Raw)

Definition: Raw observations and measurements from the work itself.

Characteristics:

  • Not processed or analyzed
  • Direct measurements
  • Automatically generated

Examples:

  • Actual cost incurred
  • Actual duration spent
  • Number of defects found
  • Percentage complete

2. Work Performance Information (Processed)

Definition: Result of analyzing and interpreting data.

Characteristics:

  • Data that has been processed
  • Analyzed and interpreted
  • Provides meaning

Examples:

  • Cost variance (actual cost vs budget)
  • Schedule variance
  • CPI and SPI calculations
  • Trends and patterns

3. Work Performance Reports (Contextualized)

Definition: Data and information presented in context for decision-making.

Characteristics:

  • Formatted for stakeholders
  • Provides context
  • Enables decisions
  • Stakeholder-friendly

Examples:

  • Status reports
  • Progress dashboards
  • Variance reports
  • Forecasts and predictions

üí° Exam Memory Aid

Raw ‚Üí Analyzed ‚Üí Contextualized

  • Data: "The project cost $100K" (measurement)
  • Information: "We're $20K over budget" (analysis)
  • Report: "Cost overrun requires corrective action: fast-track or extend deadline" (context for decisions)

üìù Change Requests - Four Types üî•üî•üî•

A change request is a formal proposal to modify any document, deliverable, or baseline.

‚ö†Ô∏è When Change Requests Are Required

Change requests are ONLY needed when changes affect baselines:

  • Schedule baseline (timeline)
  • Cost baseline (budget)
  • Scope baseline (deliverables)

If a change has NO impact on these baselines: No change request needed - handle within team!

1. Corrective Action

Purpose: Realign project to the plan when it has gone off track

Example Scenario: Project is delayed by 5 days

Corrective Action Options:

  • Change request to extend deadline by 5 days
  • Change request to increase budget (crash schedule with more resources)
  • Change request to reduce scope (fast-track)

Another Example: Team did out-of-scope work

  • Change request to accommodate scope into project
  • OR reverse the work (also needs change request for budget/schedule)

2. Preventive Action

Purpose: Avoid the project from going out of plan

Key Difference from Corrective:

  • Corrective: Project already off track ‚Üí fix it
  • Preventive: Project might go off track ‚Üí prevent it

Example: Weather forecast shows delays coming ‚Üí preventive action to adjust schedule now

3. Defect Repair

Purpose: Fix errors or quality issues

  • Correct defects in deliverables
  • Address quality problems
  • Rework to meet requirements

4. Updates

Purpose: Modify documents or plans based on new information

  • Update project documents
  • Revise subsidiary plans
  • Adjust processes or procedures

üîÑ Change Requests in Agile

‚ö†Ô∏è Agile Projects: Different Change Management

For Scope:

  • NO change requests for scope
  • No scope baseline exists (scope is dynamic)
  • Just create new user stories
  • Refine the product backlog

For Schedule/Cost:

  • Generally NO change requests mid-project
  • Must work within schedule and budget ceilings
  • EXCEPTION: At tail end of project
    • Can extend budget/schedule ceiling
    • OR initiate a new release (best practice)

Why This Makes Sense:

  • Can't have activity-based schedule when scope is dynamic
  • Can't have detailed itemized budget when scope is dynamic
  • Would constantly be updating - defeats Agile purpose
